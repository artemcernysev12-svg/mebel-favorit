V40_57
Дата: 12.05.2026

ЧТО ИЗМЕНЕНО
============

Убран прокси images.weserv.nl. Картинки теперь грузятся напрямую с
GitHub Pages (artemcernysev12-svg.github.io/catalog-favorit1).

Раньше так:
  Браузер → weserv.nl (Cloudflare, NL) → github.io → картинка
Теперь так:
  Браузер → github.io → картинка


ПОЧЕМУ ПРИШЛОСЬ УБРАТЬ ПРОКСИ
==============================

1. БЛОКИРОВКА В РОССИИ
   images.weserv.nl и его зеркало wsrv.nl хостятся на Cloudflare.
   В РФ Cloudflare периодически блокируется на уровне провайдеров.
   У клиентов без VPN получались белые карточки — браузер не мог
   достучаться до прокси, и весь сайт выглядел сломанным.

2. КЭШ ПРОКСИ ДЕРЖАЛ СТАРЫЕ ФОТО НЕДЕЛЯМИ
   При замене фото в репозитории по тому же пути weserv продолжал
   отдавать старую закэшированную версию. Ctrl+Shift+R не помогал —
   браузер запрашивал свежую копию у прокси, а прокси отдавал
   тот же закэшированный результат.

3. ЛИШНИЙ ХОП ЧЕРЕЗ НИДЕРЛАНДЫ
   Даже когда прокси работал, каждая картинка шла через серверы
   в Европе, добавляя задержку. github.io (через CDN Microsoft Azure
   с точками в РФ) географически ближе.


КАК ЭТО РАБОТАЕТ ТЕПЕРЬ
========================

В коде:

  function imgU(src, w){
    return src || '';
  }

Раньше эта функция возвращала URL вида:
  https://images.weserv.nl/?url=artemcernysev12-svg.github.io%2F...&w=400&output=webp&q=82

Теперь — просто исходный URL:
  https://artemcernysev12-svg.github.io/catalog-favorit1/items/4155668774/01.jpg

Логика галерей, ленивой загрузки, шаблоны карточек — ВСЕ остались как
были. imgU() вызывается в 5 местах и для каждого теперь просто
возвращает оригинальный путь.


ПРЕДВАРИТЕЛЬНОЕ СЖАТИЕ ФОТО (требуется ДО деплоя!)
====================================================

Чтобы переход не замедлил сайт, ВСЕ ФОТО в репозитории catalog-favorit1
должны быть предварительно сжаты. До V40_57 прокси на лету уменьшал
любые фото до нужного размера и конвертировал в WebP. Теперь браузер
получает то, что лежит на диске — поэтому фото нужно ужать заранее.

Параметры рекомендуемого сжатия (через скрипт compress_photos.py):
  - максимальная ширина: 1600 пикселей
  - формат: JPEG (для совместимости со всеми браузерами)
  - качество: 82 (визуально неотличимо от оригинала)

Ожидаемая экономия: репозиторий сократится с ~3.8 ГБ до ~1 ГБ.

ВАЖНО: V40_57 НЕЛЬЗЯ выкатывать ДО того, как фото в репозитории
catalog-favorit1 будут сжаты. Иначе сайт начнёт грузить оригинальные
тяжёлые фото (по 1-3 МБ каждое) и будет очень медленно работать на
мобильных.

Правильный порядок деплоя:
  1) Запустить compress_photos.py на всех фото
  2) Запушить сжатые фото в catalog-favorit1, дождаться обновления
     GitHub Pages
  3) Только после этого выкатить V40_57 в основной репозиторий сайта

ЧТО ПРОВЕРИТЬ ПОСЛЕ ДЕПЛОЯ
============================

В режиме инкогнито (чтобы обойти кэш), БЕЗ VPN:

1. Открыть https://мебель-фаворит.рф/
   - Главная страница должна нормально нарисоваться.
   - В hero-блоке справа должна быть мозаика фото.
   - Плитки категорий должны показывать превью.
2. Открыть любую карточку товара.
   - Главное фото должно прогрузиться за 1-3 секунды.
   - Превью под главным фото должны прогрузиться.
   - Клик по превью должен переключать главное фото.
3. F12 → Network → отфильтровать по image. URL картинок должны
   начинаться с https://artemcernysev12-svg.github.io/, а НЕ с
   https://images.weserv.nl/.
4. Заменить любое фото в репозитории catalog-favorit1 и сделать
   push. Через 1-2 минуты (GitHub Pages пересоберёт) новая фотка
   должна показываться на сайте без агрессивного сброса кэша.
   (Браузер всё ещё кэширует, но при первом заходе или
   Ctrl+Shift+R увидишь свежую версию.)


ЕСЛИ КАРТИНКИ ВЫГЛЯДЯТ ПЛОХО ПОСЛЕ СЖАТИЯ
==========================================

Можно перегнать compress_photos.py с более высоким качеством:
  python compress_photos.py /path/to/catalog-favorit1 --quality 88

Но это нужно делать на оригиналах с флешки, потому что сжатые фото
уже потеряли часть информации, и второй прогон только ухудшит их.


ДАЛЬНЕЙШИЕ УЛУЧШЕНИЯ
=====================

В следующих итерациях можно:

1. Сделать <picture>-разметку с WebP-вариантами.
   Сейчас все фото — JPEG. Если положить рядом .webp файлы и обернуть
   <img> в <picture><source srcset="01.webp" type="image/webp">...,
   современные браузеры будут грузить ~30% меньше. Старые увидят jpg.

2. Добавить srcset с разными размерами.
   Сейчас все фото 1600px независимо от устройства. Можно положить
   рядом 800px-версии и через srcset мобильные будут грузить мелкие.

3. Включить полноценный service worker с кэшированием фото.
   После первого визита картинки берутся из кэша устройства, второй
   заход моментальный даже на 3G.
