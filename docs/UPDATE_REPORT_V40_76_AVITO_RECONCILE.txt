ИСТОРИЯ БАГА (для протокола)
----------------------------
В прошлых версиях каталога часть ID хранилась в укороченном
виде из-за того, что при импорте Excel где-то по дороге
числовые значения теряли цифры на концах:
  • Одну ведущую (8059052075 → 059052075)
  • Две — ведущую и хвостовую (8058917410 → 05891741)
  • Иногда даже три (8027090800 → 0270908)

Это касалось:
  • 232 ID Микон-карточек (из 254 ВСЕХ Микон-карточек!).
  • 14 ID Эра-карточек (шкафы Дуэт/Трио + один София).

К тому моменту, когда я взял каталог на анализ, поля `img`
и `avito` (URL объявления) уже указывали на правильные
10-значные ID, но сам `id` карточки был укороченным.
Поэтому:
  • Фотки на GitHub Pages лежали правильно (под полными ID).
  • При клике "Перейти на Авито" пользователь попадал куда нужно.
  • НО при попытке открыть карточку по правильному ID
    (например, через sitemap) — сайт показывал 404.
  • И связки комплектов в kitchen-comp.js не находили модули
    в каталоге, потому что ссылались на правильные ID, а
    карточки лежали под укороченными.

РЕКОМЕНДАЦИЯ НА БУДУЩЕЕ
------------------------
Источник бага — где-то в скрипте, который генерирует catalog.js
из Авито-XLSX. Когда будешь в следующий раз пересобирать
catalog.js, выполни:
  1. Сгенерировать catalog.js (любым способом).
  2. python3 validate-catalog.py /путь/к/Catalog.xlsx
  3. Если хоть один битый ID — НЕ ДЕПЛОИТЬ, найти и поправить.

Альтернатива: попроси меня (или того, кто пишет генератор)
явно приводить ID к строке перед записью, например:
  ad_id = str(int(row['Номер объявления на Авито']))
вместо позволять Excel/JSON делать неявное преобразование float→int.
